# Postmortem: v0.2.0 live-progress sidebar artifact

**Status:** Rolled back to v0.1.1 on 2026-05-24. **§9 hypothesis RESOLVED = STREAMING PASS on 2026-05-25.** v0.5.0 ships subagent-side artifact streaming as the production live-progress channel — see §12 below for the closure note.
**Author:** Grigorii Moskalev (orchestration), Claude (implementation).
**Iterations consumed:** 6 dist builds (v0.1.2 → v0.1.6 probe, v0.2.0 implementation), then 1 probe dist (v0.5.0-probe.zip) that resolved §9.

---

## 1. Summary

v0.2.0 shipped a Cowork artifacts (`mcp__cowork__create_artifact` + `update_artifact`) "live-progress" channel on the premise that artifact-update tool-indicator strips render in chat scroll in real time during a plugin-skill orchestrator turn — bypassing the well-known Cowork text-buffering bug. **Real-run E2E in the user's `/legal-memo-writer:memo` task on 2026-05-24 falsified that premise.** The artifact strips AND the sidebar "Live artifacts" card BOTH buffered to end-of-turn at the Phase 7.5 source-review checkpoint, alongside the assistant text and Progress blocks. The only mid-turn signal during the long block was the three researcher `Agent` dispatch tiles — exactly what already worked in v0.1.1.

The probe sequence that led to v0.2.0 misread its own results. The deciding probe (v0.1.6) ran a ~5-second assistant turn whose end-of-turn flush was triggered by the closing `AskUserQuestion` modal — that batched flush was misinterpreted as mid-turn streaming because the strips appeared sequentially over those few seconds. The smoke test never actually demonstrated mid-turn streaming under production-realistic conditions (multi-minute turns with intermixed Agent dispatches), but the design that flowed from it assumed it had.

v0.2.0 is rolled back; v0.1.1 is restored as the stable baseline. The tool-surface enumeration and the encoding gotcha discovered during the probe sequence remain empirically correct and are preserved in the project memory (`reference_cowork_live_progress.md`) for the next attempt.

## 2. The original problem

The legal-memo-writer pipeline has long autonomous blocks where the chat goes quiet for 5–40 minutes:
- **Phase 5 → Phase 7.5:** parallel research (3 researchers) → sufficiency review → currency check → source-pack assembly → source-review checkpoint.
- **Phase 8 → Phase 12:** memo-writer drafting → revision loop (up to 3 iterations × 5 reviewers + mediator + writer) → client-readiness review → optional polish → docx export.

Root cause: Cowork's chat renderer only flushes assistant text on three documented triggers (end-of-assistant-turn / user input / specific side-surface tool calls — TodoWrite is one). Between flush events, queued assistant text and modals stay invisible to the user. See GitHub issues #26805, #29773, #29547, #33564, #44776 — all closed without upstream fix.

Prior mitigations on the legal-memo-writer side:
- **v0.0.42:** Mandatory `TodoWrite` side-panel updates as a secondary live channel. `mcp__ccd_session__mark_chapter` dividers at phase boundaries. Per-researcher TodoWrite sub-items in Phase 5 so the user can see "3 researchers running" in the side panel even when chat is silent.
- **v0.0.43:** Phase 7.5 rewritten as an EXPLICIT end-of-turn (`source_review_pending`) — Cowork's only documented mid-pipeline flush trigger. Text-parsed `continue` / `cancel` instead of AskUserQuestion (which was found to silently fail post-parallel-Agent in plugin-skill context).
- **v0.0.44:** Removed all downstream AskUserQuestion gates (Phase 9 iteration gates, Phase 9 forced-exit gate, Phase 10 polish gate) for the same silent-fail reason. The pipeline runs Phase 8 → Phase 12 autonomously in one turn after `continue` at source-review.

After v0.0.44, the user's experience during a `/memo` run was:
- Phase 1 → Phase 4 (intake / mode / plan): multiple end-of-turn flushes punctuated by AskUserQuestion gates that DO work pre-parallel-Agent.
- Phase 5 → Phase 7.5: single long autonomous turn. Chat shows 3 researcher tiles + 3 more agent tiles (sufficiency, currency, source-pack-builder) but no assistant text until the Phase 7.5 explicit end-of-turn flush dumps the full Progress audit trail at once.
- Phase 8 → Phase 12 (after user `continue`): a second long autonomous turn, 15–40 minutes of visual silence punctuated by ~10 subagent tiles (writer, reviewers, mediator, client-readiness reviewer).

User asked for live progress *inside* the chat surface — not just the right-side TodoWrite panel — during these long blocks. v0.2.0 was the attempt at that.

## 5. The real-run failure (2026-05-24)

User ran `/legal-memo-writer:memo "We're rolling out an AI support feature that uses customer chat transcripts. Do we need a separate legal basis under GDPR, or does our existing contract basis cover it? Also: DPIA, AI Act classification."`

Observable behaviour:
- **Phase 1 (setup):** precheck ran clean. The three `mcp__cowork__*` tools were discoverable. `state.json.config.live_progress_enabled = true`. Master artifact `memo-20260524T141725Z-...-live` was minted via `create_artifact` — the chat indicator strip for the create call appeared at the moment Phase 1 ended (i.e. at the next AskUserQuestion / end-of-turn that came shortly after).
- **Phase 2 (intake), Phase 1.5 (mode), Phase 4 (plan approval):** several short turns, each ending in an AskUserQuestion or text-parsed gate. The `update_artifact` strips for each phase transition appeared at end-of-turn alongside the Progress blocks — same flush moment as the assistant text. Consistent with v0.1.1 behaviour.
- **Phase 5 → Phase 7.5 (the long turn):** assistant turn spanned several minutes with parallel researcher dispatch, sufficiency review, currency check, source-pack assembly. During this turn the user reports:
  > «Так это ничего не поменяло, artefact заапдейтился только когда уже пользователю предложили continue (опять все появилось вместе, до этого только 3 агента выбли видны)».
  - i.e. "this changed nothing; the artifact only updated when the user was offered continue; again everything appeared together; before that only the 3 agents were visible."
- **Sidebar "Live artifacts" card:** not mid-block. Confirmed via follow-up `AskUserQuestion` this turn: «Нет, sidebar тоже обновился только в конце (с chat flush)».

Net delta vs v0.1.1: zero from the user's chat-scroll point of view. The extra `Updated artifact: <task_id>-live` strips simply appeared inside the same end-of-turn flush that was already producing all the Progress blocks. The sidebar card materialised at the same moment.

The fundamental assumption behind v0.2.0 — that `update_artifact` is a side-surface tool call analogous to `TodoWrite` that flushes immediately — was falsified.

## 6. What we learned (now ground truth)

| Mid-turn action by the orchestrator | Visible mid-orchestrator-turn? |
|---|---|
| Assistant text between tool calls | ✗ Buffered to end-of-turn. |
| `mcp__cowork__update_artifact` chat strip | ✗ Buffered to end-of-turn (real-run confirmed). |
| "Live artifacts" sidebar card refresh | ✗ Buffered to end-of-turn (real-run confirmed). |
| `Agent` / `Task` dispatch tile (the moment the dispatch fires) | ✓ Tile renders immediately in chat scroll. |
| `TodoWrite` side-panel update | ✓ Side panel only — NOT chat scroll. |

The only two confirmed mid-orchestrator-turn flush mechanisms are `Agent` dispatch (chat scroll) and `TodoWrite` (side panel). Everything else queues until the next end-of-turn / user-input / side-surface-tool trigger.

Independent of streaming, the `mcp__cowork__*` tool surface map from v0.1.5 is still empirically correct:

| Tool | Behaviour |
|---|---|
| `mcp__cowork__create_artifact` | First-write-wins. Same `id` errors. Requires `id` (kebab-case) + `html_path` (absolute path to a pre-written HTML file) + `description`. |
| `mcp__cowork__update_artifact` | Replaces the HTML content of an existing artifact in place. Requires `id` + `html_path` + `update_summary`. Silent in plugin-skill context (no approval modal blocks the call). |
| `mcp__cowork__list_artifacts` | Discovery. Useful for resume logic. |
| `mcp__cowork__allow_cowork_file_delete` | File-level (not artifact-level) deletion permission. |
| `mcp__cowork__read_widget_context` | Reads interactive widget state. |
| `mcp__cowork__request_cowork_directory` | Requests cowork directory access. |
| No `delete_artifact`, no `chat_message`, no `notify`. |

The encoding gotcha is real and surfaced again during the smoke probes:
- HTML files served via `create_artifact` / `update_artifact` MUST include `<meta charset="UTF-8">` in `<head>`. Cowork's artifact iframe does not auto-detect UTF-8 — em-dashes, curly quotes, and Cyrillic render as mojibake (`â€"`) without the meta tag. `scripts/render_live_progress.py` (now deleted with the rollback) handled this correctly; future implementations should reuse the same `<meta charset="UTF-8">` discipline.

The tool surface absences are also still real:
- `Monitor` — not in plugin-skill (only in main session).
- `PushNotification` — not in plugin-skill (only in main session).
- `mcp__ccd_session__mark_chapter` / `spawn_task` — not in plugin-skill ToolSearch even though `progress-contract.md` references them.

## 7. Root cause of the misread

The v0.1.6 probe asked the user whether 1 × `create_artifact` + 3 × `update_artifact` + `AskUserQuestion` produced "3 cards updating in place / 3 separate cards / 1 unchanged card / something else". The user answered "State D, no approval prompts" and described the strips appearing sequentially over ~3 seconds.

The interpretation written into v0.2.0's design was: "update_artifact streams live, no approval blocks, ergo it bypasses Cowork's text-buffering bug."

The interpretation that fits BOTH the v0.1.6 observation AND the v0.2.0 real-run observation: the `AskUserQuestion` at the end of v0.1.6 triggered an end-of-turn flush (modals are explicit flush triggers). Cowork rendered all 4 strips inside that flush moment. The strips appeared sequentially because each `update_artifact` call has ~500ms of network/render latency and they were dispatched back-to-back, so the visual cascade looked like streaming. **The smoke test never actually demonstrated mid-turn streaming during a long quiet block.** It only demonstrated that, when an end-of-turn flush happens, all queued strips get rendered.

The probe needed to include explicit waits to falsify the "streams during the turn" hypothesis. For example: 1 × `create_artifact` + `sleep 30` + 1 × `update_artifact` + `sleep 30` + 1 × `update_artifact` + end of turn. Under the "streams live" hypothesis the user would see the second card update at the 30-second mark; under the "buffers to end-of-turn" hypothesis they would see both updates pile up only at the closing flush. Neither v0.1.4 nor v0.1.6 had any waits between calls.

**Lesson: a probe that doesn't replicate the production turn length cannot validate mid-turn behaviour.** A flush-on-end behaviour and a true streaming behaviour are indistinguishable inside a 5-second turn. Future probes for streaming claims MUST include explicit waits (e.g. `sleep 20` between tool calls) or otherwise simulate production turn duration before drawing conclusions.

A secondary lesson: when a smoke test produces a "yes" answer that would unblock significant implementation work, the test is high-stakes and deserves a deliberate falsifiability check. The v0.1.6 question to the user was framed as "did you see what I predicted?" — which is the wrong framing. A better framing would have been "describe exactly when the strips appeared relative to the question modal opening" — that wording would have caught the end-of-turn batching.

## 9. Open question for the next iteration

> «а нельзя в работу субагентов что-то добавить чтобы там обновлялось и выдавало в чат?»

The user's direction: surface live progress from inside subagents' work, not from the orchestrator.

The unanswered question that determines whether subagent-driven streaming is viable:

> **Do `mcp__cowork__update_artifact` calls made from inside a dispatched subagent (i.e. by the subagent itself during its own work, not by the orchestrator) bypass the main-orchestrator-turn buffer and surface chat-scroll strips in real time as the subagent works?**

The orchestrator is blocked between `Agent` dispatch and return — its own tool calls buffer to end-of-turn. But subagents have their own turn lifecycle. In principle, their tool indicators could flush at their own rhythm, independent of the parent orchestrator's buffer state. This is empirically untested.

**Suggested next probe (not yet built):** a one-off skill that dispatches a single subagent. The subagent's prompt instructs it to: `create_artifact` for id `probe-subagent-stream` → wait 5 seconds (Bash `sleep 5`) → `update_artifact` step 1 → wait 5 seconds → `update_artifact` step 2 → wait 5 seconds → `update_artifact` step 3 → return. Total subagent runtime ~20 seconds. The orchestrator does nothing else during this time. If the user sees the four strips appear at ~5-second intervals as the subagent works, subagent-side streaming is viable. If they all appear at the moment the subagent returns to the orchestrator, the chat-scroll signal granularity is "one flush per subagent return" (better than "one flush per orchestrator turn" but worse than "one flush per milestone within a subagent").

Resume of live-progress work is conditional on a "yes" answer to that probe. If "no", the alternative paths are: (a) fragment the heaviest single-subagent phases (Phase 8 memo-writer is the prime candidate) into per-issue mini-subagent dispatches so each return is a chat flush; (b) accept the limitation and double down on the TodoWrite side panel as the live-progress channel.

## 12. Closure — §9 resolved 2026-05-25 (STREAMING PASS)

The §9 hypothesis (subagent-side `mcp__cowork__update_artifact` calls bypass the orchestrator-turn buffer and surface in the parent chat scroll in real time as the subagent works) was tested empirically via `legal-memo-writer-0.5.0-probe.zip` on 2026-05-25 and resolved **STREAMING PASS**.

### Probe procedure

- A diagnostic skill `/legal-memo-writer:probe subagent-artifact` dispatched a single `probe-subagent-streamer` subagent whose prompt instructed it to: mint an artifact via `create_artifact` → `sleep 25` → `update_artifact` step 1 → `sleep 25` → `update_artifact` step 2 → `sleep 25` → `update_artifact` step 3 → return. Total subagent runtime ≈ 80 seconds. The orchestrator did nothing during this window — any chat-surface event during those ~80 seconds was unambiguously attributable to the subagent's tool calls.
- The 25-second sleeps were the falsifiability lock. The v0.1.6 misread happened because that probe was ~5 seconds long and a closing `AskUserQuestion` modal triggered an end-of-turn flush that visually mimicked streaming. With 25-second gaps, a true streaming behavior is visually unambiguous from an end-of-turn batch flush.
- The probe HTML (preserved at `dist/legal-memo-writer-0.5.0-probe.zip` → `agents/probe-subagent-streamer.md`) included `<meta charset="UTF-8">` as required by Cowork's iframe — the encoding gotcha from the original probe remains binding for production HTML.

### Result

The user reported (verbatim): «Это работает, обновляется в режиме реального времени артефакт, это удобно и классно выглядит, если еще бы задизайнить этот html красиво». Translation: "It works, the artifact updates in real-time mode, it's convenient and looks great, only if we still designed this HTML beautifully." The "Update artifact" tool-indicator strips appeared in the chat scroll AND the sidebar "Live artifacts" card refreshed at ~25-second intervals as the subagent worked — not batched at subagent return, not buffered to end-of-orchestrator-turn.

### Additional finding from the same probe — stronger AskUserQuestion silent-fail

The probe's AskUserQuestion modal (which was supposed to capture the §9 result formally via four pre-baked options) **never rendered.** The orchestrator showed "Using AskUserQuestion... · 2m 36s" indefinitely. This extends the v0.0.43 documented finding ("AskUserQuestion silently fails post-parallel-Agent in plugin-skill context"): single-Task dispatch is sufficient to break the next AskUserQuestion call — the failure is not specific to parallel dispatch. Operational implication: ALL gates after ANY `Task(...)` dispatch must use text-parsed gates (the Phase 7.5 mechanism), not AskUserQuestion. The probe skill's design forgot this — the AskUserQuestion at the end of `probe subagent-artifact` was the wrong instrument; the §9 result was instead captured via direct user report.

### v0.5.0 architecture

Subagent-side artifact streaming is the production live-progress channel:

- **Master artifact** is minted in `skills/memo/SKILL.md` Phase 1 step 3.5 via `mcp__cowork__create_artifact`. ID is `memo-<task_id>-live`; html_path is `<work_dir>/live-progress.html`. State.json captures both.
- **Phase-boundary updates** by the orchestrator (orchestrator-side calls still buffer to end-of-turn — that is acceptable; subagents' own calls provide the in-phase live signal).
- **Subagent-side updates** at internal step boundaries by every heavy / medium subagent: `memo-writer` (per-issue), 3 researchers (per-issue), 5 reviewers (start + done), `revision-mediator` (start + per-iteration verdict), `client-readiness-reviewer`, `research-sufficiency-reviewer`, `currency-checker`, `source-pack-builder`, `fact-assumption-analyst`. Each subagent reads `state.json.config.live_progress_enabled` and skips silently if false (best-effort posture).
- **Canonical contract** at `skills/memo/references/live-progress-contract.md`. Renderer at `scripts/render_live_progress.py` (32 unit tests). Encoding rule from v0.2.0 (`<meta charset="UTF-8">`) preserved.

### v0.5.0 does NOT ship

- Multi-turn architecture via `ScheduleWakeup` (was the alternative in the design plan at `C:\Users\User\.claude\plans\c-users-user-desktop-cowork-claude-memo-inherited-gizmo.md`). Deferred — subagent-side streaming proved sufficient. The plan remains on disk for future revisit.
- Per-issue Phase 8 fragmentation (Branch B from the design plan). Not needed because subagent-side updates from `memo-writer` give per-issue live progress within a single subagent dispatch.
- The `legal-memo-writer:probe` diagnostic skill — removed from the production v0.5.0 dist (preserved in `dist/legal-memo-writer-0.5.0-probe.zip` as historical archive).

### Wasted vs preserved

**Wasted from this postmortem cycle:** the v0.1.6 probe misread + v0.2.0 implementation (~4 hours) remain wasted as documented in §8. But the §9 hypothesis itself was the right next question — confirming it correct closes that arc.

**Preserved and now active in v0.5.0:**
- The `mcp__cowork__create_artifact` / `update_artifact` tool surface enumeration.
- The `<meta charset="UTF-8">` encoding discipline.
- The atomic write contract (renderer uses `.tmp` + rename).
- The "best-effort, never block the pipeline on live-progress" philosophy.
- The institutional memory in this postmortem.

This postmortem is now closed.
