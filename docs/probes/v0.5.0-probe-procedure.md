# Probe procedure — v0.5.0-probe

**Audience:** developer (Grigorii). Not for end users.
**Goal:** answer four empirical questions about Claude Code / Cowork tool-surface behavior in plugin-skill context, so we can commit to (or kill) the multi-turn architecture before writing any production-skill code.

The questions, in priority order:

| ID | Question | Probe |
|---|---|---|
| V1 | Is `ScheduleWakeup` available in plugin-skill context when invoked via `/loop`? | `multiturn` step 1 |
| V2 | Does the wake fire in the **same** chat scroll (not a new conversation)? | `multiturn` step 2 |
| V3 | What's the actual wake delay vs the documented 60s floor? | `multiturn` step 2 |
| §9 | Do subagent-side `update_artifact` calls surface in the parent chat scroll as the subagent executes them, or only at subagent return / end-of-orchestrator-turn? | `subagent-artifact` |

V4 / V5 / V6 from the plan are NOT covered by this probe build. They can be answered in a follow-up probe iteration once V1–V3 + §9 are settled.

---

## 0. Install the probe dist

1. Make sure the file exists: `legal-memo-writer/dist/legal-memo-writer-0.5.0-probe.zip` (60 files, ~312 KB).
2. In Cowork: Settings → Plugins → drag-and-drop the `0.5.0-probe.zip` file. It will replace the currently installed `0.4.0` build.
3. Verify install: open any Cowork chat and type `/legal-memo-writer:` — autocomplete should show the four production skills (`memo`, `continue`, `status`, `style`) plus `probe` (the new diagnostic).

If autocomplete does NOT show `probe`, the install failed. Check for backslash entries (shouldn't happen — the build script verified) or a plugin-loader error in Cowork's logs.

## 1. Probe A — `subagent-artifact` (no `/loop` needed, run this first)

This is the easier probe and doesn't depend on `/loop` mode being viable. Run it first so you have at least one data point if anything goes sideways with the wake probe.

### Steps

1. Open a fresh Cowork chat (so there's no clutter in the scroll).
2. **Before** typing the command, open the Cowork sidebar "Live artifacts" panel — keep it visible throughout the run. You will watch the panel during the ~80-second subagent run.
3. Type and submit:

   ```
   /legal-memo-writer:probe subagent-artifact
   ```

4. The skill prints a heads-up block explaining the timing (mint at t=0, updates at t≈25/50/75s, return at t≈80s). It then dispatches the `probe-subagent-streamer` subagent via a `Task` tool call. **You should see exactly ONE chat tile appear for that dispatch** — the standard Agent dispatch tile, which is a confirmed mid-turn flush mechanism.
5. **Watch for ~80 seconds.** During this time the orchestrator does NOTHING — no text, no tool calls. Any chat-surface event you see is from the subagent's tool calls.
   - Pattern (a) — STREAMING: you see one "Updated artifact" strip and one sidebar card refresh every ~25 seconds, four events total spread across the 80s.
   - Pattern (b) — BATCHED at subagent return: nothing visible for ~80s, then all four events (the create strip + 3 update strips, and the final sidebar card content) appear together at t≈80s when the subagent return tile renders.
   - Pattern (c) — FULLY BUFFERED: nothing visible until the AskUserQuestion modal appears after the subagent return. The strips arrive in the same flush as the modal.
6. After ~80s, the subagent returns its summary block and the skill prints it back to you. The skill calls `mcp__cowork__list_artifacts` (sanity check) and then renders an `AskUserQuestion` modal.
7. **Answer the modal honestly based on what you actually observed.** The question wording is deliberately about timing, not about confirming a prediction — the v0.1.6 misread happened because the probe asked "did you see what I predicted" rather than "describe what you saw". Pick the option that matches reality. If none of the four pre-baked options fit, use the free-form Other to describe.

### Interpretation

- **Selected option 1 (STREAMING):** §9 = PASS. The next production release can layer subagent-side `update_artifact` calls onto the heaviest subagents (`memo-writer`, researchers) to give inside-phase visibility. Combine with multi-turn (if V1–V3 also pass) for full coverage.
- **Selected option 2 (BATCHED at subagent return):** §9 = PARTIAL. Subagent-side calls don't stream mid-subagent, but at least they flush at subagent return — which is BETTER than the orchestrator-side v0.2.0 behavior (which buffered to end-of-orchestrator-turn). Could still be useful if the heavy subagents are themselves short (~30s each), but for memo-writer (~5-10 min) the user still sees nothing during the subagent's own run.
- **Selected option 3 (FULLY BUFFERED):** §9 = FAIL. Same failure as v0.2.0. Subagent-side artifacts contribute nothing. Drop this path entirely.
- **Selected option 4 (PARTIAL) or 5 (Other):** report the free-form text verbatim; needs case-by-case analysis.

### Where the result is recorded

`~/.claude/plugin-data/legal-memo-writer/probe-state/subagent-artifact-results.log` — one entry per probe run with the user's selected option and elapsed timing.

## 2. Probe B — `multiturn` (requires `/loop` invocation)

This probe answers V1 / V2 / V3 in a single wake cycle: initial step prints time, schedules wake, ends turn → wake fires ~60s later → step 2 prints elapsed time and asks user about chat continuity.

### Steps

1. Make sure no `multiturn` state file exists from a prior run. If unsure, first run:

   ```
   /legal-memo-writer:probe reset
   ```

   (No `/loop` prefix needed for reset.) This deletes the probe state directory.

2. **Important: use the `/loop` prefix.** Without it, `ScheduleWakeup` is not available in plugin-skill ToolSearch, and V1 will trivially fail (which would just mean "you forgot /loop", not that the architecture is dead). Type and submit:

   ```
   /loop /legal-memo-writer:probe multiturn
   ```

3. **Step 1 of the probe.** The skill calls `ToolSearch(query="select:ScheduleWakeup")`. If the schema returns, V1 = PASS, and the skill proceeds to:
   - Print step-1 announcement with the current absolute time.
   - Write a state file marking step 2 next.
   - Call `ScheduleWakeup(60s, prompt="/loop /legal-memo-writer:probe multiturn", reason=...)`.
   - End its turn.

   If V1 fails (schema doesn't load), you get a "V1 FAIL" message — abort the probe, document the failure, and report back. This means either /loop doesn't expose ScheduleWakeup in plugin-skill context (architecture is dead from this surface) or there's some other surface-availability issue.

4. **Do NOTHING for ~60–90 seconds.** Do not type messages in the chat. Do not click anything in this conversation. Note the absolute time you saw step 1 end. Observe whether anything happens in the chat surface during the wait.

5. **Step 2 — the wake fires.** After ~60s (per the documented floor), the same `/loop /legal-memo-writer:probe multiturn` should auto-fire and the skill resumes. **Critical observation:** where does step 2 appear?
   - **Same chat scroll, directly below step 1, no manual action:** V2 = PASS.
   - **New conversation / sidebar item / separate surface:** V2 = FAIL.
   - **Same chat but only after you focused / clicked / typed something:** V2 = PARTIAL.
   - **Step 2 never appears at all even after 5+ minutes:** the wake silently didn't fire — separate failure mode, report verbatim.

6. The step-2 announcement prints V3 (actual delay in seconds) and asks for V2 confirmation via `AskUserQuestion`. **Answer based on what you actually observed in step 4.** Do not pick the "PASS" option just because it's the desired outcome.

### Interpretation

- **V1 PASS + V2 = same chat + V3 in [55s, 90s]:** multi-turn architecture is viable. Commit to v0.5.0 implementation.
- **V1 PASS + V2 = different chat:** multi-turn architecture fragments UX. Abandon — or pivot to one of the out-of-frame channels (file-as-display, browser dashboard, etc.).
- **V1 PASS + V2 = needs user action:** quasi-autonomous. Document the friction, decide if it's still worth shipping.
- **V1 PASS + V2 PASS + V3 > 90s consistently:** wake fires reliably but slowly. Each phase boundary adds more wallclock overhead than the plan estimated. Re-cost the architecture; may still ship.
- **V1 FAIL even with /loop:** ScheduleWakeup is plugin-skill-restricted in a way the schema description doesn't admit. Abandon multi-turn. Report to Anthropic.
- **Wake never fires:** any of (a) Cowork closed and didn't requeue on next launch, (b) plugin-skill /loop is silently broken, (c) some other surface failure. Investigate before drawing conclusions.

### Where the result is recorded

`~/.claude/plugin-data/legal-memo-writer/probe-state/multiturn-results.log`

## 4. Re-running probes

To re-run a probe (after changing something or to confirm a result):

```
/legal-memo-writer:probe reset
```

…then re-invoke the probe action. The state directory is wiped; the next run starts from step 1 again.

The `*-results.log` files are **appended to**, never wiped — they keep history across all probe runs. This is intentional so you can compare results from multiple runs (e.g. checking V3 wake-delay consistency across 5 successive runs).

---

## 5. Rolling back to 0.4.0

When you're done with probes, install `legal-memo-writer-0.4.0.zip` (still in `dist/`) to restore the production behavior. The probe state files at `~/.claude/plugin-data/legal-memo-writer/probe-state/` are not part of the plugin install — they survive the rollback and remain available as historical record. Delete them manually if you want a clean slate:

```
rm -rf "$HOME/.claude/plugin-data/legal-memo-writer/probe-state"
```

The probe skill and `probe-subagent-streamer` agent disappear from the plugin's slash-command autocomplete and dispatch list once 0.4.0 is reinstalled — they exist only in the 0.5.0-probe zip.

## 6. Known limitations of this probe build

- **V4 (non-/loop wake prompts):** not tested. Whether `ScheduleWakeup(prompt="/legal-memo-writer:continue <task_id>", ...)` fires correctly when the original /loop prompt was different is an open question. If V1–V3 pass, V4 determines the implementation choice: does each wake re-fire the original `/loop /legal-memo-writer:memo "<query>"` (requires task-discovery on re-entry) or can it directly target `/legal-memo-writer:continue` (simpler)? Add a V4 probe in a follow-up build if needed.
- **V5 (user types during wake window):** not tested. Probably the user typing a new message just cancels the pending wake (typical /loop dynamic-mode behavior), but unverified.
- **V6 (Cowork closed between wakes):** not tested. The `mcp__scheduled-tasks` schema says "runs on next launch if app was closed when due"; whether `ScheduleWakeup` has the same guarantee is unknown.
- **Subagent dispatch synchronicity during multi-turn:** the probe doesn't dispatch subagents inside the `multiturn` probe. If subagent dispatches behave differently across wake boundaries (e.g. in-flight subagent at wake fire time), that's also unverified.

These limitations don't block the v0.5.0 commit decision — V1+V2+V3+§9 are sufficient to choose the path. The remaining V4/V5/V6 details can be settled during implementation by adding small smoke tests inside the v0.5.0 build itself.
